# 功能详解六：AI 智能助手

> AI 助手是系统的亮点功能。用户可以用自然语言询问仓储相关问题，系统调用 DeepSeek 大模型来回答。

## 二、前端实现

### 2.1 聊天记录的存储

```javascript
// 聊天记录存储在 localStorage 中
// Key 格式：ai_chat_msg_<用户名>_<会话ID>
const CHAT_STORAGE_PREFIX = 'ai_chat_msg_'
this.storageKey = CHAT_STORAGE_PREFIX + this.username + '_' + this.sessionId

// 发送消息后保存
localStorage.setItem(this.storageKey, JSON.stringify(this.messages))

// 页面加载时读取
const saved = localStorage.getItem(this.storageKey)
if (saved) this.messages = JSON.parse(saved)
```

**为什么按会话隔离？** 每次登录生成新的 `session_id`，这样不同登录会话的聊天记录不会混在一起。退出登录时清除当前会话的聊天记录。

### 2.2 发送消息

```javascript
async sendMessage() {
  if (!this.inputText.trim()) return

  // 1. 把用户消息添加到列表中
  this.messages.push({ role: 'user', content: this.inputText })

  // 2. 发送请求到后端 AI 接口
  const res = await request.post('/ai/chat', {
    question: this.inputText,           // 当前问题
    history: this.messages.slice(0, -1) // 历史对话（用于上下文）
  })

  // 3. 把 AI 的回复添加到列表中
  this.messages.push({ role: 'assistant', content: res.data })

  // 4. 保存到 localStorage
  localStorage.setItem(this.storageKey, JSON.stringify(this.messages))

  // 5. 清空输入框，滚动到底部
  this.inputText = ''
}
```

**关键点**：前端不只发送当前问题，还把**历史对话**一起发给后端。这样 AI 才能理解上下文。比如用户先说"查一下冷冻鸡排的库存"，然后说"那出库记录呢？"，AI 需要知道"那"指的是"冷冻鸡排"。

### 2.3 从主页快捷跳转

```javascript
// 主页的AI输入框
this.$router.push({ name: 'AiChat', query: { q: this.aiQuestion } })

// AI页面检测URL参数
mounted() {
  const q = this.$route.query.q
  if (q) {
    this.inputText = q
    this.sendMessage()  // 自动发送
  }
}
```

## 三、后端实现（核心！）

### 3.1 Controller 层

```java
@PostMapping("/chat")
public ResultData chat(@RequestBody Map<String, Object> params) {
    String question = (String) params.get("question");
    List<Map<String, String>> history = (List) params.get("history");

    // 调用 Service 获取 AI 回复
    String reply = aiService.chat(question.trim(), history);

    return ResultData.success(reply);
}
```

非常简单，只是把参数传给 Service。

### 3.2 Service 层（AiServiceImpl）—— 真正的逻辑

```java
public String chat(String userMessage, List<Map<String, String>> conversationHistory) {
    // 步骤1：构建消息列表（发给 DeepSeek 的格式）
    List<Map<String, Object>> messages = new ArrayList<>();

    // 步骤2：添加"系统提示词"——告诉AI它是谁，能做什么，不能做什么
    messages.add({ role: "system", content: SYSTEM_PROMPT });

    // 步骤3：注入真实数据库数据作为上下文
    String dataContext = buildDataContext(userMessage);
    if (!dataContext.isEmpty()) {
        messages.add({ role: "system", content: dataContext });
    }

    // 步骤4：添加历史对话（限制最近 10 轮，防止超出 token 限制）
    // ...

    // 步骤5：添加用户当前问题
    messages.add({ role: "user", content: userMessage });

    // 步骤6：用 OkHttp 发送 HTTP 请求到 DeepSeek API
    Request request = new Request.Builder()
        .url("https://api.deepseek.com/v1/chat/completions")
        .post(RequestBody.create(json, MediaType.parse("application/json")))
        .addHeader("Authorization", "Bearer " + apiKey)
        .build();

    // 步骤7：解析 DeepSeek 返回的 JSON，提取回复内容
    Response response = okHttpClient.newCall(request).execute();
    return extractContent(response.body().string());
}
```

### 3.3 系统提示词（SYSTEM_PROMPT）

这是整个 AI 功能中最重要的设计。提示词定义了 AI 的"人设"：

```
你是一名专业的预制菜仓储管理(WMS)智能助手，严格遵守以下所有规则：

1、仅回答与预制菜新增批次、出入库、库存盘点、库存调整、质检、
   处理次品等相关问题；

2、绝对拒绝回答娱乐、游戏、金融、政治、个人生活等任何非仓储
   物流及食品安全类问题；

3、不提供具体的菜谱、烹饪方法或菜品口味评价；

4、对于系统报错、数据异常、设备故障等紧急情况，必须明确提示
   用户立即联系技术支持或现场主管处理；

5、回答专业准确，逻辑清晰，使用规范的仓储物流术语（如SKU、
   批次号、先进先出FIFO等）；

6、如果用户提问无关问题，统一回复："抱歉，我仅能提供预制菜
   仓储管理相关的咨询服务..."

7、【严禁编造数据】你绝对不能编造、猜测或虚构任何数字。如果
   用户询问具体数据而你无法获取，必须明确告知用户去对应页面查看。

8、所有回答末尾必须添加免责声明。
```

### 3.4 真实数据注入（buildDataContext）

这是本系统 AI 助手的**一大亮点**。普通的 AI 聊天不知道你仓库里有什么数据，但这个系统会根据用户的问题，自动查询数据库并注入真实数据：

```java
private String buildDataContext(String userMessage) {
    StringBuilder ctx = new StringBuilder();
    ctx.append("【以下是当前WMS系统的真实数据，回答时请引用这些数据，不要编造】\n");

    // 如果用户问题包含"库存"关键词 → 查 inventory 表，注入库存数据
    if (containsAny(userMessage, "库存", "存货", "数量", "还有多少")) {
        ArrayList<Inventory> invList = inventoryMapper.list();
        // 统计各种货物的库存数量
        ctx.append("- 库存总览：共 ").append(totalItems).append(" 条记录...\n");
        // 按货物名称汇总
        for (entry : goodsSummary) {
            ctx.append("  · ").append(name).append("：").append(qty).append("件\n");
        }
    }

    // 如果用户问"货物" → 查 goods 表
    if (containsAny(userMessage, "货物", "商品", "有哪些货")) {
        // ...注入货物列表
    }

    // 如果用户问"仓库" → 查 warehouse 表
    if (containsAny(userMessage, "仓库", "库位", "空位")) {
        // ...注入仓库和可用库位信息
    }

    // 如果用户问"批次" → 查 batch 表
    // 如果用户问"任务" → 查 work_task 表
}
```

**简单理解**：用户问"现在库存有多少？" → 后端先去数据库查真实数据 → 把数据附在提示词里 → DeepSeek 基于真实数据回答，而不是瞎编。

## 四、完整数据流

```
用户输入："冷冻鸡排还有多少库存？"
  │
  ▼
前端发送 POST /ai/chat
  Body: { question: "冷冻鸡排还有多少库存？", history: [...] }
  │
  ▼
后端 AiController.chat()
  │
  ▼
AiServiceImpl.chat()
  │
  ├── 分析用户问题，发现包含"库存"关键词
  ├── 调用 inventoryMapper.list() 查询真实库存数据
  ├── 调用 goodsMapper.findById() 关联货物名称
  ├── 构建上下文：冷冻鸡排 200件...
  │
  ├── 组装消息：
  │   1. 系统提示词（定义AI人设）
  │   2. 真实数据上下文（冷冻鸡排库存200件...）
  │   3. 历史对话
  │   4. 用户当前问题
  │
  ├── 用 OkHttp 发送给 DeepSeek API
  │    https://api.deepseek.com/v1/chat/completions
  │
  ├── DeepSeek 返回：
  │    "根据系统数据，冷冻鸡排当前库存为200件，分布在3个库位中..."
  │
  └── 解析回复，返回给前端
  │
  ▼
前端收到回复，显示在聊天界面
```

## 五、你可能被问到的问题

**Q: AI 助手用的是哪个大模型？怎么接入的？**

A: 使用 DeepSeek（深度求索）的 chat 模型。通过 OkHttp 发送 HTTP POST 请求到 `https://api.deepseek.com/v1/chat/completions`，携带 API Key 进行身份验证。请求体遵循 OpenAI 兼容的格式，包含 messages 数组（system prompt + 上下文数据 + 历史对话 + 用户问题）。

**Q: AI 如何知道仓库的真实数据？**

A: 后端在调用 DeepSeek 之前，先分析用户问题中的关键词（如"库存"、"货物"、"仓库"），然后查询对应的数据库表获取真实数据，把这些数据作为"系统消息"注入到给 DeepSeek 的提示词中。DeepSeek 看到真实数据后，基于这些数据回答，而不是编造。

**Q: 为什么要加系统提示词？**

A: 不加提示词，AI 就是一个通用聊天机器人，可能会回答娱乐、政治等不相关的问题，也可能编造数据。系统提示词限制了 AI 的行为边界，让它只回答仓储管理相关问题，并且要求它不能编造数据。

**Q: AI 聊天记录存在哪里？丢了怎么办？**

A: 存在浏览器的 localStorage 中。如果清除浏览器数据或换一台电脑，聊天记录会丢失。这是前端的简化方案，更完善的做法是把记录存到后端数据库。

**Q: 为什么要把历史对话发给 AI？**

A: 因为 HTTP 请求是无状态的。如果不发历史对话，AI 每次只能看到当前这一句话，无法理解上下文。比如用户先问"冷冻鸡排有多少？"，然后问"那什么时候入库的？"，如果不传历史，AI 不知道"那"指的是什么。
