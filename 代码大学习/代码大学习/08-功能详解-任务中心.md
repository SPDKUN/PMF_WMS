# 功能详解五：任务中心（最核心、最复杂的模块）

> 任务中心是整个 WMS 系统的核心，涵盖了仓库作业的 8 种操作。这是整个项目中代码最多、业务逻辑最复杂的部分。

## 二、我的待办（Todo List）

### 获取待办任务

```
GET /workTask/myTasks?assigneeId=<当前用户ID>
```

后端查询所有 `assignee_id = 当前用户` 且 `completed_time IS NULL`（未完成）的工作任务。

### 任务表格展示

| 列名 | 来源字段 | 说明 |
|------|---------|------|
| 单号 | related_order_no | 关联的业务单号（如 I20260715001） |
| 批次号 | related_batch_id | 关联的批次号 |
| 类型 | task_type | 入库/出库/盘点/质检/调整/次品处理 |
| 优先级 | priority | 高/中/低（用不同颜色标签显示） |
| 截至日期 | deadline | 任务需要在什么时候完成 |
| 操作 | — | "去完成"按钮 |

### 点击"去完成"

每个任务类型对应不同的完成表单。例如：
- 入库任务 → 弹出选择库位的对话框
- 出库任务 → 弹出确认出库的对话框
- 质检任务 → 弹出填写质检结果的表单
- 盘点任务 → 弹出录入实际数量的表单

## 四、核心操作一：货物入库（最详细的示例）

入库是系统中最复杂的操作之一。我们以它为例，详细讲解完整流程。

### 4.1 创建入库任务

**前端操作**：在操作中心点击"货物入库" → 弹出表单 → 选择批次 → 填写信息 → 点击"确认创建"

**前端发送的请求**：
```javascript
POST /api/inbound/create
Body: {
  "batchId": "B20260711003",    // 选择要入库的批次
  "inboundType": "采购入库",      // 入库类型
  "priority": "高",              // 优先级
  "operatorId": 1,               // 当前用户ID
  "deadline": "2026-07-16",     // 截止日期
  "remark": "紧急入库"            // 备注
}
```

**后端做了什么**（InboundServiceImpl.createInboundTask）：

```java
@Transactional  // ★ 事务注解：下面的操作要么全成功，要么全失败
public WorkTask createInboundTask(CreateInboundRequest request) {
    // 步骤1：生成入库单号
    // 格式：I + 日期 + 序号（如 I20260715001）
    String inboundNo = "I" + today + String.format("%03d", maxSeq + 1);

    // 步骤2：插入入库单头（inbound_order_head）
    // 记录：单号、类型、状态="草稿"、操作人

    // 步骤3：插入入库单明细（inbound_order_detail）
    // 记录：货物ID、批次号、生产日期、到期日、数量

    // 步骤4：创建工作任务（work_task）
    // 记录：类型="入库"、关联单号、分配给操作人

    // 步骤5：记录操作日志（operation_log）
    // 记录：谁在什么时候做了什么操作
}
```

### 4.2 为什么要用 @Transactional（事务）？

假设步骤 1-3 成功了，但步骤 4 时数据库突然挂了：

- **没有事务**：入库单头、明细已经写入，但工作任务没创建。数据不一致！
- **有事务**：Spring 自动回滚，前面 3 步的操作全部撤销。数据库回到操作前的状态。

**事务就是"要么全做，要么全不做"的保证。**

### 4.3 完成入库任务

入库任务创建后，出现在仓管员的"我的待办"中。仓管员点击"去完成"，选择库位，完成入库。

**前端发送的请求**：

```javascript
PUT /api/inbound/complete/5    // 5 是任务ID
Body: {
  "locationIds": [10, 11, 12],  // 选择的3个库位
  "operatorId": 1,
  "remark": "已入库"
}
```

**后端做了什么**（InboundServiceImpl.completeInbound）：

```java
@Transactional
public void completeInbound(Integer taskId, CompleteInboundRequest request) {
    // 1. 查询工作任务和批次信息
    // 2. 验证：库位数量是否符合要求（每10件占一个库位，向上取整）
    //    → 比如 25 件货物，需要 3 个库位
    // 3. 删除旧库存（"待入库"状态）
    // 4. 遍历每个库位：
    //    a. 标记库位为"已占用"
    //    b. 创建库存记录（状态改为"正常"）
    //    c. 更新库区和仓库的可用库位数
    // 5. 更新商品总数量
    // 6. 更新批次状态为"正常"
    // 7. 更新入库单头状态为"已完成"
    // 8. 完成工作任务
    // 9. 记录操作日志
}
```

### 4.4 库位分配的计算逻辑

```java
// 每10件货占1个库位，向上取整
int totalNeeded = (int) Math.ceil(initialQty / 10.0);

// 例如：25 件 → ceil(2.5) = 3 个库位
// 前2个库位各放10件，最后1个库位放5件
for (int i = 0; i < locationIds.size(); i++) {
    int qty = (i < totalNeeded - 1) ? 10 : (initialQty - (totalNeeded - 1) * 10);
    // 创建库存记录
}
```

### 4.5 拒绝入库

如果入库单有问题，可以拒绝：

```java
PUT /api/inbound/reject/5
Body: { "operatorId": 1 }
```

后端操作：
1. 删除入库单明细
2. 删除入库单头
3. 删除工作任务
4. 记录操作日志

## 五、其他 7 种操作简述

### 5.1 出库（OutboundController + OutboundServiceImpl）

与入库相反的操作流程：
- 创建出库任务 → 选择要出库的库存 → 生成出库单 → 完成任务
- 出库完成后释放库位，更新仓库可用库位数

### 5.2 库存调整（AdjustmentController + AdjustmentServiceImpl）

调整货物的库位或数量：
- 记录从哪个库位移到哪个库位
- 更新库存和库位占用状态

### 5.3 库存盘点（InventoryCheckController + InventoryCheckServiceImpl）

核对实际库存和系统记录：
- 创建盘点任务 → 录入实际数量 → 对比差异
- 记录 `book_quantity`（系统记录数量）和 `actual_quantity`（实际数量）
- 产生差异报告

### 5.4 质量检查（QualityCheckController + QualityCheckServiceImpl）

对货物进行质检：
- 质检员检查货物 → 记录检查结果（合格/不合格）
- 不合格的货物 → 记录缺陷原因和处理建议

### 5.5 次品处理（DefectiveController + DefectiveServiceImpl）

处理不合格产品：
- 报废、退货、返工等处理方式
- 更新库存状态为"待报废"

### 5.6 新增批次（BatchController + BatchServiceImpl）

创建新的货物批次：
- 选择货物、填写生产日期、到期日、初始数量
- 系统自动生成批次号（格式：B + 日期 + 序号，如 B20260715003）
- 批次初始状态为"待检"

### 5.7 温湿度上传（TemperatureHumidityRecordController）

记录仓库环境数据：
- 选择仓库、填写温度（℃）和湿度（%RH）
- 用于食品安全监控

## 六、所有操作涉及的数据库表

以入库操作为例，一次完整的入库涉及 **7 张表**：

```
inbound_order_head      ← 入库单头
inbound_order_detail    ← 入库单明细
work_task               ← 工作任务
operation_log           ← 操作日志
batch                   ← 批次（更新状态）
inventory               ← 库存（新建记录）
location                ← 库位（标记占用）
zone                    ← 库区（更新可用数）
warehouse               ← 仓库（更新可用数）
goods                   ← 货物（更新总数）
```

这就是为什么 Service 层需要 `@Transactional` —— 一张表操作失败，所有表都需要回滚。


## 八、你可能被问到的问题

**Q: 入库时为什么要锁定库位？**

A: 防止多个操作同时使用同一个库位。库位被占用后 `lock_status` 变为"锁定"，其他入库操作就无法选择这个库位。出库完成后释放锁定。

**Q: `@Transactional` 事务是什么？为什么要用它？**

A: 事务保证一组数据库操作"要么全部成功，要么全部失败"。入库操作涉及 7-9 张表的更新，如果其中一步失败，前面已执行的操作需要回滚，否则会导致数据不一致。Spring 的 `@Transactional` 注解自动实现了这个机制。

**Q: 入库单号 I20260715001 是怎么生成的？**

A: 前缀 I（Inbound）+ 当前日期（yyyyMMdd）+ 三位流水号。系统先查当天已有的单号，找到最大流水号，然后 +1。

**Q: 待办任务列表是怎么判断哪些任务属于当前用户的？**

A: 查询 `work_task` 表中 `assignee_id = 当前用户ID` 且 `completed_time IS NULL` 的记录。

**Q: 货物入库时，25 件货物为什么需要 3 个库位？**

A: 系统规定每个库位最多放 10 件。`Math.ceil(25/10) = 3`，前 2 个库位各放 10 件，最后 1 个放 5 件。这个 10 件的限制是硬编码在后端代码中的业务规则。
