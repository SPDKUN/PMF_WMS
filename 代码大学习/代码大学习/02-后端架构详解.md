# 后端架构详解：Spring Boot 项目是如何组织的

> 读完本章，你将理解后端代码的"分层架构"，以及请求从前端发过来后，在后端经历了怎样的处理过程。

---

## 一、分层架构（最重要的一张图）

后端代码采用了经典的**三层架构**：

```
┌──────────────────────────────────────────────────┐
│  浏览器/前端                                       │
│  发送 HTTP 请求: GET /api/user/personnel           │
└──────────────────┬───────────────────────────────┘
                   │ HTTP请求到达后端 8088 端口
                   ▼
┌──────────────────────────────────────────────────┐
│  第一层：Filter（过滤器）                           │
│  JwtAuthenticationFilter                          │
│  职责：检查请求有没有合法的 JWT 令牌                  │
│  如果没有令牌或令牌过期 → 直接返回错误，不往下走       │
│  如果令牌有效 → 放行，把 userId 存入 request         │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│  第二层：Controller（控制器）                       │
│  例如：UserController                             │
│  职责：接收请求参数，调用 Service，返回 JSON 响应     │
│  代码很少，只是一个"传话筒"                          │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│  第三层：Service（服务层）                          │
│  例如：UserServiceImpl                            │
│  职责：真正的业务逻辑在这里！                         │
│  比如：创建入库单要同时操作多张表、校验数据等           │
│  方法上可能标注 @Transactional（事务）               │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│  第四层：Mapper（数据访问层）                        │
│  例如：UserMapper.java + UserMapper.xml            │
│  职责：执行 SQL 语句，读写数据库                     │
│  每个方法对应一条 SQL                                │
└──────────────────┬───────────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────────┐
│  MySQL 数据库                                     │
└──────────────────────────────────────────────────┘
```

**为什么要分层？** 每层只做自己的事，互不干扰。Controller 不需要知道 SQL 怎么写，Mapper 不需要知道业务逻辑。这样代码好维护、好测试。

## 二、关键代码文件逐个讲解

### 2.1 项目入口：PmfApplication.java

```java
@SpringBootApplication  // ← 这个注解告诉Spring Boot："我是主程序"
public class PmfApplication {
    public static void main(String[] args) {
        SpringApplication.run(PmfApplication.class, args);  // 启动整个项目
    }
}
```

位置：`backend/PMF/src/main/java/com/neuedu/pmf/PmfApplication.java`

这就是整个后端的入口。运行这个 `main` 方法，就启动了整个项目。`@SpringBootApplication` 注解触发了"自动配置"——Spring Boot 会自动扫描项目中的所有类，把它们管理起来。

### 2.2 统一返回格式：ResultData.java

后端给前端返回的数据，统一用这个格式包装：

```json
{
  "code": 200,       // 状态码：200=成功，1001=用户名密码错误，1002=令牌过期
  "msg": "操作成功",   // 提示信息
  "data": { ... },   // 真正的数据
  "token": "xxx"     // 登录时才会有的令牌
}
```

```java
public class ResultData extends HashMap<String, Object> {
    // 成功时调用
    public static ResultData success(Object data) { ... }

    // 失败时调用
    public static ResultData fail(ResultCode resultCode) { ... }
}
```

位置：`backend/PMF/src/main/java/com/neuedu/pmf/common/ResultData.java`

**这样设计的好处**：前端不管调用哪个接口，返回的格式都是一样的，解析起来很方便。

### 2.3 状态码枚举：ResultCode.java

```java
public enum ResultCode {
    SUCCESS(200, "操作成功"),
    FAILED(500, "操作失败"),
    USER_AND_PASSWORD_ERROR(1001, "用户名或密码错误"),
```

把所有可能的状态码集中定义在这里，方便管理。

### 2.5 JWT 过滤器：JwtAuthenticationFilter.java

这个过滤器**在每次请求到达 Controller 之前执行**，负责拦截请求并验证令牌。

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    // 这些路径不需要验证令牌（登录、注册、Swagger文档等）
    private static final String[] EXCLUDE_PATHS = {
        "/auth/login", "/register", "/swagger-ui/**", ...
    };

    @Override
    protected void doFilterInternal(HttpServletRequest request, ...) {
        // 1. 判断请求路径是否需要放行
        if (isExcludePath(request.getRequestURI())) {
            filterChain.doFilter(request, response);  // 直接放行
            return;
        }

        // 2. 从请求头中取出 token
        String token = jwtUtil.extractTokenFromHeader(
            request.getHeader("Authorization"));

        // 3. 验证 token 是否有效
        if (!jwtUtil.validateToken(token)) {
            throw new JwtException("token无效或者过期");
        }

        // 4. 把 userId 存入 request，供后面的 Controller 使用
        Long userId = jwtUtil.getUserIdFromToken(token);
        request.setAttribute("userId", userId);

        filterChain.doFilter(request, response);  // 放行
    }
}
```

**图解**：
```
请求进来 → JWT过滤器 → 令牌有效？→ 是 → 放行到Controller
                      → 否 → 返回{"code":1002, "msg":"令牌已过期"}
```

### 2.6 CORS 跨域配置：CorsConfig.java

```java
@Configuration
public class CorsConfig {
    private CorsConfiguration corsConfig() {
        CorsConfiguration config = new CorsConfiguration();
        config.addAllowedOriginPattern("*");   // 允许任何来源访问
        config.addAllowedHeader("*");          // 允许任何请求头
        config.addAllowedMethod("*");          // 允许任何HTTP方法
        config.setAllowCredentials(true);      // 允许携带cookie
        return config;
    }
}
```

**为什么需要这个？** 浏览器的"同源策略"默认禁止一个域名下的网页请求另一个域名的 API。前端在 `localhost:8080`，后端在 `localhost:8088`，不同端口也算"跨域"。CORS 配置就是告诉浏览器："允许跨域请求"。

### 2.7 全局异常处理：GlobalExceptionHandler.java

```java
@RestControllerAdvice  // 拦截所有Controller中抛出的异常
public class GlobalExceptionHandler {
    // 处理业务异常
    @ExceptionHandler(BusinessException.class)
    public ResultData handleBusinessException(BusinessException e) { ... }

    // 处理参数校验失败
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResultData handleValidationException(...) { ... }

    // 兜底处理所有没预料到的异常
    @ExceptionHandler(Exception.class)
    public ResultData handleException(Exception e) { ... }
}
```

**有了它**：Controller 里的方法就不需要写 `try-catch` 了。任何地方抛出异常，都会被这里统一处理，返回一个格式化的错误信息。

## 四、Entity（实体类）是什么？

Entity 是 Java 类和数据库表的**一一对应**。

例如，数据库有 `user` 表：

```sql
CREATE TABLE user (
    user_id INT PRIMARY KEY,
    username VARCHAR(50),
    password VARCHAR(200),
    position VARCHAR(20)
);
```

对应的 Java 类：

```java
@Data  // Lombok注解：自动生成getter/setter
public class User {
    private Integer user_id;    // 对应数据库的 user_id 列
    private String username;    // 对应 username 列
    private String password;    // 对应 password 列
    private String position;    // 对应 position 列
}
```

位置：`backend/PMF/src/main/java/com/neuedu/pmf/entity/`

本项目共有 **22 个实体类**，分别对应 22 张数据库表。

## 五、VO（值对象）是什么？

有些请求的参数很复杂，不像简单的增删改查只有几个字段。VO 就是用来**封装复杂请求参数**的。

例如，创建入库任务时前端要传很多参数：
```java
public class CreateInboundRequest {
    private String batchId;       // 批次号
    private String inboundType;   // 入库类型
    private String priority;      // 优先级
    private Integer operatorId;   // 操作人ID
    private String deadline;      // 截止日期
    private String remark;        // 备注
}
```

位置：`backend/PMF/src/main/java/com/neuedu/pmf/vo/`

## 六、一个完整的请求处理流程（以登录为例）

```
1. 用户在前端输入用户名"admin"和密码"123456"，点击登录

2. 前端发送 GET 请求：/api/auth/login?username=admin&password=123456

3. 请求到达后端 8088 端口
   → JwtAuthenticationFilter 检查：/auth/login 在放行列表中，直接放行

4. 请求到达 AuthController.login() 方法
   → 调用 authService.login("admin", "123456")

5. AuthServiceImpl.login() 执行：
   a. 根据用户名在数据库中找到用户记录
   b. 用 BCrypt 比对密码是否匹配
   c. 如果匹配，生成 JWT 令牌（包含 userId=1, username="admin"，2小时有效）
   d. 把用户信息缓存到 Redis
   e. 返回 User 对象

6. 回到 AuthController：
   → ResultData.success(user) + 放入 token
   → 返回 JSON：{"code":200, "msg":"操作成功", "data":{用户信息}, "token":"eyJ..."}

7. 前端收到响应：
   → 把 token 存入 localStorage
   → 把用户信息存入 localStorage
   → 跳转到主页
```
