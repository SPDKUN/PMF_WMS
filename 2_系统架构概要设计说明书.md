# 系统架构概要设计说明书——预制菜WMS仓储管理系统

## 1. 引言

### 1.1 项目背景

预制菜WMS智能仓储管理系统是为预制菜生产企业量身打造的仓库管理平台。系统覆盖从货物入库、存储管理、质量检查、出库发货到库存盘点的全生命周期，采用 **B/S + 微信小程序** 架构模式，实现前后端分离，支持Web管理端和微信小程序移动端双平台访问。

### 1.2 设计目标

- **业务闭环**：实现批次创建 → 入库 → 库存管理 → 质检 → 出库 → 报表分析的完整业务链路
- **多端覆盖**：一套后端同时服务Web端(Vue 3)和微信小程序端
- **智能化**：接入DeepSeek大语言模型，实现AI辅助仓储管理
- **数据可视化**：通过ECharts/Canvas图表实时展示仓储运营数据
- **安全可靠**：JWT认证 + BCrypt密码加密 + AES-256传输加密 + 事务保护

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| WMS | Warehouse Management System，仓库管理系统 |
| SKU | Stock Keeping Unit，库存量单位 |
| 库位 | 仓库中的最小存储单元(货架格位) |
| 批次 | 同一批生产的货物，用批次号标识 |
| FIFO | First In First Out，先进先出原则 |
| JWT | JSON Web Token，无状态身份认证令牌 |

### 3.3 统一响应格式

所有API返回统一JSON结构 `ResultData extends HashMap<String, Object>`:

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": { ... },
  "token": "eyJhbG..."
}
```

状态码定义 (`ResultCode`枚举):
| Code | 含义 |
|------|------|
| 200 | 操作成功 |
| 500 | 操作失败 |
| 1001 | 用户名或密码错误 |
| 1002 | 令牌已过期或失效 |
| 1003 | 业务处理异常 |

### 3.4 JWT认证流程

```
用户登录 → POST /auth/login
  ↓
AuthServiceImpl: BCrypt验证密码 → JwtUtil.generateToken(userId, username)
  ↓
返回 { user, token } → 前端存localStorage
  ↓
后续请求: Authorization: Bearer <token>
  ↓
JwtAuthenticationFilter.doFilterInternal()
  → 白名单路径放行 (/auth/login, /register, /swagger-ui/**)
  → 其他路径: 提取token → 验证有效性 → 注入userId到request
  → token无效: 返回 {"code":1002, "msg":"令牌已过期"}
```

JWT配置: 密钥签名(HMAC-SHA384), 2小时过期, 载荷包含userId和username.

### 3.5 全局异常处理

`GlobalExceptionHandler` (@RestControllerAdvice):
- `BusinessException` → 业务异常(1003)
- `MethodArgumentNotValidException` → 参数校验失败
- `Exception` → 兜底处理(500)

## 7. 核心业务流程设计

### 7.1 入库流程

```
创建批次(B) → 创建入库单(I) → 审核通过 → 员工执行入库
  → 选择库位(按储存条件过滤) → 确认执行
  → 系统操作(10张表):
    1. inbound_order_head    — 状态→已完成
    2. inbound_order_detail  — 记录明细
    3. work_task             — 标记完成
    4. operation_log         — 记录操作
    5. inventory             — 创建库存记录(状态→正常)
    6. location              — 标记库位占用
    7. zone                  — 减可用库位
    8. warehouse             — 减可用库位
    9. goods                 — 加库存数量
   10. batch                 — 状态→正常
```

### 7.2 出库流程

```
创建出库单(O) → 审核通过 → 员工执行出库
  → 确认出库
  → 系统操作:
    1. inventory   — 扣减库存(quantity - n)
    2. location    — 库存清空则释放库位
    3. zone/warehouse — 恢复可用库位
    4. goods       — 减总数量
    5. batch       — 减remaining_quantity
                    (归零则级联删除: batch → QC详情/盘点详情/出库详情/库存)
```

### 7.3 质检流程

```
创建质检单(C) → 质检员执行 → 选择结果
  → 合格: batch.status = '正常'
  → 不合格: batch.status = '报废', inventory.status = '待报废'
```

### 7.4 核心设计模式

**单据编号规则**：前缀 + yyyyMMdd + 3位流水号
- I → 入库单 (Inbound)
- O → 出库单 (Outbound)
- C → 质检单 (Check)
- P → 盘点单 (check)
- A → 调整单 (Adjustment)
- D → 次品处理单 (Defective)
- B → 批次 (Batch)

**库位锁定机制**：出库/调整操作期间锁定源库位，防止并发冲突。
`location.lock_status = '锁定'`, `location.lock_purpose = '出库'/'调整'`

**库位分配算法**：每库位存放10个单位，最后剩余部分放最后一个库位。
`所需库位数 = ceil(初始数量 / 10)`

## 8. AI助手架构

### 8.1 技术方案

```
用户输入问题
  ↓
POST /ai/chat { question, history }
  ↓
AiServiceImpl.chat()
  ├─ 构建系统提示词 (定义AI人设, 限制回答范围)
  ├─ buildDataContext(question)  ← 关键词匹配, 查询真实数据库
  │   ├─ "库存" → Inventory表
  │   ├─ "货物" → Goods表
  │   ├─ "仓库" → Warehouse表
  │   ├─ "批次" → Batch表
  │   └─ "任务" → WorkTask表
  ├─ 组装消息: system_prompt + data_context + history + question
  └─ OkHttp → DeepSeek API (https://api.deepseek.com/v1/chat/completions)
  ↓
返回AI回复 (基于真实数据)
```

### 8.2 系统提示词

- 定义角色："预制菜仓储管理(WMS)智能助手"
- 限制范围：仅回答仓储相关问题
- 严禁编造数据：要求引用注入的真实数据
- 安全边界：拒绝娱乐/金融/政治等无关问题
- 免责声明：每个回复末尾附加

## 9. 安全架构

### 9.1 认证安全

| 层级 | 方案 | 说明 |
|------|------|------|
| 密码存储 | BCrypt单向加密 | 不可逆，登录时比对密文 |
| 密码传输 | AES-256-CBC | 前端加密后传输，Base64编码 |
| 会话管理 | JWT (HMAC-SHA384) | 2小时过期，无状态 |
| 缓存会话 | Redis | 30分钟TTL，减少DB查询 |

### 9.2 权限安全

| 层级 | 方案 | 说明 |
|------|------|------|
| Web路由 | `router.beforeEach` | 检查token + position |
| Web菜单 | `visibleMenuItems` | 主管才显示管理菜单 |
| Web按钮 | `isAdmin`/`isStaff`/`isInspector` | 条件渲染 |
| 小程序 | `wx:if` 条件渲染 | 基于userInfo.position |
| 后端 | (当前仅前端校验) | 待加强 |

### 9.3 数据安全

- **事务保护**：`@Transactional`保证多表操作的原子性
- **CORS**：`CorsConfig`配置跨域白名单(当前开发阶段全允许)
- **SQL注入防护**：MyBatis参数化查询 `#{param}`

## 11. 技术决策说明

| 决策 | 选择 | 理由 |
|------|------|------|
| 前端框架 | Vue 3 Options API | 学习曲线平缓，适合课程项目 |

| 后端框架 | Spring Boot 3 | 生态成熟，自动配置减少样板代码 |
| ORM | MyBatis (XML) | 复杂SQL灵活控制，动态SQL支持好 |
| 缓存 | Redis | 用户会话缓存，降低DB压力 |
| AI接入 | DeepSeek (OpenAI兼容) | 性价比高，中文能力强 |
| 小程序 | 原生框架 | 避免额外学习成本，直接使用微信API |
| 构建 | Maven + Vite | 主流选择，社区支持好 |
